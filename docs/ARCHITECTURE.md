# Dotfiles v2 Architecture Draft

This document proposes an incremental v2 layout that makes the repo easier to extend without breaking the current bootstrap flow.

## Goals

- Treat each managed tool as an app with a single home.
- Keep tracked config separate from machine-specific state.
- Make installation declarative via per-app manifests.
- Support gradual migration from the current `bootstrap/tasks/*.sh` approach.
- Make it obvious how to add a new app in the future.

## App Contract

Each app should own everything needed to understand and manage it:

- `manifest.yaml`: install metadata, targets, checks, notes
- `files/`: tracked dotfiles or config files
- `hooks/`: app-specific post-install or validation scripts when needed

## Manifest Shape

The first version should stay intentionally small.

```yaml
name: git
description: Git configuration
platforms:
  - linux
  - wsl
  - macos
install:
  linux:
    packages:
      - git
  macos:
    brew:
      - git
link:
  strategy: stow
checks:
  - command: git --version
notes:
  - Creates ~/.gitconfig.local if missing
```

Recommended fields:

- `name`
- `description`
- `platforms`
- `install`
- `link`
- `checks`
- `notes`

Optional fields:

- `depends_on`
- `hooks`
- `state_paths`
- `local_overrides`

## Linking Strategy

Prefer one linking strategy across the repo.

### Recommended default: stow-compatible app folders

Each app's `files/` directory should mirror the target layout under `$HOME`.

Examples:

- `apps/zsh/files/.zshrc`
- `apps/nvim/files/.config/nvim/init.lua`
- `apps/git/files/.config/git/gitconfig`

This keeps the future migration path simple whether the repo uses GNU Stow directly or a custom symlink wrapper.

## What Stays in `config/`

The existing `config/` tree can remain during migration.

Short term:
- Keep current bootstrap tasks working.
- Add new apps under `apps/`.
- Migrate old apps one at a time.

Long term:
- Move tracked app config from `config/` into `apps/<app>/files/`.
- Keep `bootstrap/` focused on orchestration only.

## Machine State vs Tracked Files

Tracked repo files should never be generated by install steps.

Use this rule:

- `apps/*/files/**` = committed and edited by humans
- `$HOME/.local/state/**` = generated runtime state
- `*.local` or untracked files = private machine-specific overrides

Examples:

- SSH keys belong in `~/.ssh`, never in the repo.
- `~/.gitconfig.local` stays machine-local.
- Package installs and downloaded binaries should not rewrite tracked config files.

## Migration Plan

### Phase 1: Foundation

- Add `docs/ARCHITECTURE.md`
- Add `docs/ADDING_APPS.md`
- Add `scripts/add-app`
- Keep the current `bootstrap/tasks/*.sh` flow unchanged

### Phase 2: New apps use the v2 pattern

- Any brand-new app gets created under `apps/<app>/`
- Bootstrap can continue using task scripts until a manifest runner exists

### Phase 3: Manifest-aware bootstrap

Enhance `bootstrap/install.sh` to support:

- `--list-apps`
- `--only app1,app2`
- manifest validation
- generic checks

### Phase 4: Migrate existing apps

Recommended order:

1. git
2. zsh
3. tmux
4. nvim
5. optional tooling like yazi, nvm, python, dotnet

## Design Principles

- Prefer data over bespoke shell logic.
- Keep hooks as exceptions, not the default pattern.
- Keep file placement predictable.
- Make adding a new app a one-command workflow.
- Favor incremental migration over a risky big-bang rewrite.
